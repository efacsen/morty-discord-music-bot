# Architecture Patterns

**Domain:** TypeScript Discord music bot (self-hosted, open-source)
**Researched:** 2026-03-29
**Overall confidence:** HIGH (current source verification + cross-referenced community patterns)

---

## Recommended Architecture

The project retains its current **layered event-driven** design with one structural change: the build pipeline introduces a `src/` → `dist/` compile step, and the setup wizard lives as a standalone Node.js script outside that pipeline.

```
discord-music-bot/
├── src/                        # TypeScript source (compiled to dist/)
│   ├── index.ts                # Bootstrap: client init, extractor reg, dynamic loaders
│   ├── types/                  # Shared type definitions (interfaces, enums)
│   │   ├── index.ts            # Barrel re-export
│   │   ├── command.ts          # CommandModule interface
│   │   ├── event.ts            # EventModule interface
│   │   └── queue.ts            # QueueMetadata interface (GuildQueue typing)
│   ├── commands/               # One file per slash command (.ts)
│   ├── events/                 # Discord.js event handlers (.ts)
│   ├── handlers/               # Interaction logic (buttonHandler.ts)
│   ├── extractors/             # discord-player extractor plugins (.ts)
│   └── utils/                  # Pure helpers: embeds, time formatting (.ts)
├── dist/                       # Compiled JS output (git-ignored, generated by tsc)
├── scripts/
│   └── setup.ts                # Cross-platform setup wizard (Node.js script)
├── docker/
│   ├── Dockerfile
│   └── docker-compose.yml
├── tsconfig.json               # TypeScript compiler config
├── .env.example
└── package.json                # "type": "module", scripts: build/dev/start
```

### Why This Structure

The current JS structure (`commands/`, `events/`, `handlers/`, `extractors/`, `utils/`) is already sound and well-matched to discord.js patterns. The TypeScript migration does not restructure — it overlays types. The only additions are:

- `src/types/` — a dedicated directory for shared interfaces, preventing circular imports
- `dist/` — compiled output; `node` runs `dist/index.js` in production
- `scripts/setup.ts` — isolated from `src/` because it is not part of the bot runtime

## Component Boundaries

| Component | Responsibility | Communicates With | Boundary Rule |
|-----------|---------------|-------------------|---------------|
| `src/index.ts` | Bootstrap and wire-up only — no business logic | All components (reads/registers them) | Must not contain command or player logic |
| `src/types/` | Shared contracts (interfaces, enums) | Imported by all other components | No runtime code; type declarations only |
| `src/commands/` | Handle slash command interactions | discord-player via hooks (`useMainPlayer`, `useQueue`), `src/utils/` | No direct voice/audio manipulation |
| `src/events/` | Discord.js lifecycle events | `src/handlers/`, `client.commands` Collection | Thin dispatch layer only |
| `src/handlers/` | Button interaction logic | discord-player (`useQueue`), `src/utils/` | Only interacts with queue, not client directly |
| `src/extractors/` | yt-dlp audio pipeline | discord-player (`BaseExtractor`), Node `child_process` | No Discord.js imports; audio concerns only |
| `src/utils/` | Pure Discord payload builders | `discord.js` embed/button builders only | Stateless pure functions; no side effects |
| `scripts/setup.ts` | One-time setup wizard | OS shell (execa/cross-spawn), Discord REST API | Must never import from `src/`; self-contained |

**Critical boundary:** `src/extractors/YtDlpExtractor.ts` must not import from `src/commands/` or `src/events/`. The extractor layer communicates only upward through discord-player's event system — never directly.

## Data Flow

### Play Command (user invocation to audio stream)

```
Discord User
  → /play <query>
  → [Discord Gateway WebSocket]
  → interactionCreate event (src/events/interactionCreate.ts)
  → client.commands.get('play').execute(interaction)      [command dispatch]
  → player.search(query)                                  [discord-player]
  → YtDlpExtractor.validate() + handle()                 [extractor]
  → yt-dlp --dump-json <url>                             [child_process]
  → Track[] returned to discord-player
  → queue.addTrack(track) + queue.node.play()            [queue mutation]
  → willPlayTrack event → resolver() called              [CRITICAL: unblocks pipeline]
  → YtDlpExtractor.stream(track)                        [extractor]
  → yt-dlp audio bytes → PassThrough(32MB)              [stream buffer]
  → mediaplex Opus encoding
  → @snazzah/davey DAVE encryption
  → Discord voice channel
  → playerStart event → createPlayerEmbed() → channel.send() [embed]
```

**Key flow constraint:** The `willPlayTrack` resolver must be called by `src/index.ts`. This wiring stays in the bootstrap file because it is a player-level concern that spans all queue operations.

### Button Interaction

```
Discord User → clicks button (customId: player_* or queue_*)
  → interactionCreate event
  → if customId.startsWith('player_') or 'queue_' → handlePlayerButton(interaction)
  → useQueue(guildId) → queue mutation
  → createPlayerEmbed() / createQueueEmbed() → message.edit()
```

### State Location

| State | Where It Lives | Type |
|-------|---------------|------|
| Per-guild audio queue | discord-player `GuildQueue` in-memory | Runtime only |
| Now Playing message refs | `Map<string, NowPlayingMessage>` in `src/index.ts` module scope | Runtime only |
| Bot configuration | `process.env` (loaded via dotenv) | Boot-time only |
| yt-dlp cookie auth | `process.env.YTDLP_COOKIES_FILE` | Passed to extractor at stream time |

## TypeScript Layer Design

### Shared Types (`src/types/`)

Three interfaces are required by multiple components and must be defined centrally to avoid circular imports:

**`command.ts`**
```typescript
import { SlashCommandBuilder, ChatInputCommandInteraction } from 'discord.js';

export interface CommandModule {
  data: SlashCommandBuilder | Omit<SlashCommandBuilder, 'addSubcommand' | 'addSubcommandGroup'>;
  execute: (interaction: ChatInputCommandInteraction) => Promise<void>;
}
```

**`event.ts`**
```typescript
export interface EventModule<T extends keyof ClientEvents = keyof ClientEvents> {
  name: T;
  once?: boolean;
  execute: (...args: ClientEvents[T]) => Promise<void>;
}
```

**`queue.ts`**
```typescript
import { TextChannel, VoiceChannel } from 'discord.js';

export interface QueueMetadata {
  channel: TextChannel;
  voiceChannel: VoiceChannel;
}
```

The `QueueMetadata` interface is the most critical: it types `GuildQueue<QueueMetadata>` so that `queue.metadata.channel` has a known type throughout commands and handlers, eliminating the `any` casts that would otherwise be needed.

**Confidence: HIGH** — discord-player ships TypeScript definitions and its `GuildQueue` is a generic: `GuildQueue<T>` where `T` is the metadata type.

### Build Configuration

**Recommended `tsconfig.json`:**
```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "rootDir": "./src",
    "outDir": "./dist",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  },
  "include": ["src/**/*.ts"],
  "exclude": ["node_modules", "dist"]
}
```

**Why `module: "NodeNext"` over `"preserve"`:**
The project already uses `"type": "module"` in package.json (ESM). `NodeNext` is the correct resolution mode for ESM Node.js projects: it enforces explicit file extensions in imports (e.g., `import './utils/formatTime.js'` — the `.js` extension must be present even though the source file is `.ts`). This is a known footgun covered below in Pitfalls.

**Confidence: HIGH** — TypeScript official docs confirm `NodeNext` is the correct pairing for Node.js ESM with `"type": "module"`.

## Build Pipeline

### Development

```
tsx watch src/index.ts
```

`tsx` uses esbuild under the hood for near-instant restarts. It handles ESM natively with zero configuration. Replaces the current `node --watch src/index.js` dev script.

**Confidence: HIGH** — tsx is the current consensus choice for TypeScript ESM watch mode in 2025-2026, confirmed by multiple sources.

### Production

```
tsc                              # type-check + emit JS to dist/
node dist/index.js               # run compiled output
```

Do not use `tsx` in production. `tsc` provides type-checking that `tsx` skips. The compiled `dist/` output is what Docker runs.

### Package Scripts

```json
{
  "scripts": {
    "build": "tsc",
    "dev": "tsx watch src/index.ts",
    "start": "node dist/index.js",
    "typecheck": "tsc --noEmit",
    "setup": "tsx scripts/setup.ts"
  }
}
```

**Why not `tsup`:** tsup is optimized for library authoring (dual ESM+CJS output, `.d.ts` bundling). For a bot application there is no consumer — no need for bundling or declaration merging. Plain `tsc` is simpler and more transparent.

## Setup Wizard Architecture

The setup wizard (`scripts/setup.ts`) is a separate concern from the bot runtime. It runs once, at install time, and must work cross-platform.

### Responsibilities (ordered)

1. **System check** — detect Node.js version, yt-dlp, ffmpeg; print install instructions if missing
2. **Env generation** — prompt for `DISCORD_CLIENT_TOKEN` and `DISCORD_CLIENT_ID`; write `.env`
3. **Dependency install** — run `npm install` if `node_modules` absent
4. **Command registration** — run `node dist/index.js` equivalent or direct REST API call to register slash commands
5. **Connection test** — validate the bot token hits Discord's `/users/@me` endpoint successfully

### Implementation Pattern

Use `@inquirer/prompts` (successor to `inquirer`) for interactive prompts and `execa` for cross-platform subprocess execution. Both are ESM-native.

```typescript
// scripts/setup.ts — self-contained, never imports from src/
import { input, password } from '@inquirer/prompts';
import { execa } from 'execa';
import { writeFile } from 'fs/promises';
```

**Do not use raw `readline`** — it requires handling of Windows CRLF and terminal color codes manually.

**Confidence: MEDIUM** — `@inquirer/prompts` is the maintained successor to `inquirer` v9+. `execa` is the standard cross-platform subprocess tool confirmed in multiple Node.js tooling guides.

## Patterns to Follow

### Pattern 1: Explicit `.js` Extensions in ESM Imports

With `module: "NodeNext"`, TypeScript requires import paths to use the runtime extension (`.js`), even though the source file is `.ts`.

```typescript
// CORRECT
import { formatDuration } from './utils/formatTime.js';

// WRONG — will fail at runtime with NodeNext resolution
import { formatDuration } from './utils/formatTime';
```

This is counterintuitive but mandatory. All imports across `src/` must include `.js` extensions.

### Pattern 2: Typed GuildQueue

Every command that accesses `queue.metadata` should type the queue with the shared `QueueMetadata` interface:

```typescript
import { useQueue } from 'discord-player';
import type { QueueMetadata } from '../types/queue.js';

const queue = useQueue<QueueMetadata>(guildId);
if (!queue) return;
queue.metadata.channel.send('...');  // typed — no any cast needed
```

### Pattern 3: Dynamic Loader with Type Assertion

The `src/index.ts` dynamic loader casts imported modules to the known interface:

```typescript
import type { CommandModule } from './types/command.js';

const files = readdirSync('./src/commands').filter(f => f.endsWith('.ts') || f.endsWith('.js'));
for (const file of files) {
  const mod = await import(`./commands/${file}`) as { default: CommandModule };
  client.commands.set(mod.default.data.name, mod.default);
}
```

In production this runs against `dist/commands/*.js`. The dynamic import path changes between `dev` (tsx, `.ts` files) and `start` (tsc output, `.js` files) — use a path resolution approach that works in both, or separate the dev and prod loaders.

**Recommended:** In production, `dist/` mirrors `src/` structure (tsc preserves directories). Load from `./commands/` relative to `dist/index.js`.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Importing from `dist/` at dev time

Do not write imports that reference `dist/` paths. The `tsx` dev workflow runs source files directly — `dist/` only exists after `npm run build`. Mixing the two causes confusing "module not found" errors.

### Anti-Pattern 2: `ts-node` with ESM

`ts-node` has known friction with Node.js ESM (`"type": "module"`), requiring `--esm` flag and extra `tsconfig` settings. Use `tsx` instead. Evidence: multiple community guides explicitly call out ts-node ESM setup as broken or requiring complex workarounds in 2025.

**Confidence: HIGH** — confirmed by tsx official docs and community comparisons.

### Anti-Pattern 3: Barrel Exports in Commands/Events

Avoid `index.ts` barrel files in `commands/` or `events/`. The dynamic loader in `src/index.ts` glob-loads all files in these directories. A barrel export would cause the loader to double-register commands. Keep each command as a standalone module.

### Anti-Pattern 4: Putting Setup Logic in `src/`

The setup wizard (`scripts/setup.ts`) must never import from `src/`. Reason: `src/` imports `discord-player` and other bot dependencies that expect a running Discord client. The setup wizard runs before the bot starts — importing bot code would instantiate player objects with no valid token.

### Anti-Pattern 5: Embedding GuildQueue metadata as `any`

```typescript
// BAD — kills type safety throughout the codebase
const queue = useQueue(guildId);
(queue?.metadata as any).channel.send('...');

// GOOD — use the typed interface
const queue = useQueue<QueueMetadata>(guildId);
queue?.metadata.channel.send('...');
```

## Build Order (Dependency Between Components)

Components must be built and typed in this order because of import dependencies:

```
1. src/types/           — no dependencies; must be defined first
2. src/utils/           — depends on discord.js types only
3. src/extractors/      — depends on discord-player BaseExtractor types
4. src/handlers/        — depends on types/ + utils/ + discord-player
5. src/commands/        — depends on types/ + utils/ + discord-player
6. src/events/          — depends on types/ + handlers/ + commands/ (via Collection)
7. src/index.ts         — depends on everything; wires it together
8. scripts/setup.ts     — independent; depends on @inquirer/prompts + execa only
```

**Implication for migration phases:** Migrate in this order. Each layer should compile cleanly before moving to the next. Start with `src/types/` (no logic, just interfaces) to establish contracts, then migrate `src/utils/` (pure functions, easy wins), then extractors, then commands, then events, then `src/index.ts`.

## Scalability Considerations

This bot is self-hosted, single-guild or small-guild focused. Scalability concerns are operational, not architectural.

| Concern | For this project | If ever needed |
|---------|-----------------|----------------|
| Multiple guilds | Per-guild GuildQueue already handles this | discord-player handles natively |
| Process crashes | `unhandledRejection` guard in index.ts | PM2 or Docker restart policies |
| yt-dlp failures | Catch in stream() + player error event | No change needed |
| Large queue memory | In-memory; fine for self-hosted | Not in scope |
